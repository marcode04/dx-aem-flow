  <div class="grid grid-cols-1 md:grid-cols-2 gap-6 mt-6">
    <ContentCard icon="mdi:router" iconColor="bg-brand-700" title="WI Router (POST /wi)">
      Handles all <code>workitem.updated</code> events. Scans work item tags against configured
      agents to determine which pipeline to queue. Single Lambda, single route, multiple agents.

      <strong>Processing flow:</strong>
      1. Validate webhook (Basic Auth + secret header)
      2. Parse payload -- extract work item ID, tags, type
      3. Scan AGENTS array -- match tag gate + work item type
      4. First matching agent wins (tags should be mutually exclusive)
      5. Deduplicate (DynamoDB, 1-hour TTL)
      6. Rate limit (DynamoDB atomic counter)
      7. Queue ADO pipeline with templateParameters
    </ContentCard>

    <ContentCard icon="mdi:chat" iconColor="bg-cyan" title="PR Router (POST /pr-answer)">
      Handles <code>git.pullrequest.comment-event</code> -- triggered when someone comments on a PR.

      <strong>Five gates:</strong>
      1. Event type includes "comment"
      2. PR is active
      3. PR is mine (createdBy matches <code>MY_IDENTITIES</code>)
      4. Comment is NOT from me (avoid loop)
      5. Comment is NOT from bot (avoid loop)
    </ContentCard>
  </div>
</Section>

<SectionHeading
  badge="Design"
  badgeColor="warning"
  title="Design Decisions"
  subtitle="Why tag-based routing and why two separate Lambdas."
  bg="alt"
/>
<Section>
  <div class="grid grid-cols-1 md:grid-cols-2 gap-6">
    <ContentCard icon="mdi:tag" iconColor="bg-brand-700" title="Why Tag-Based Routing?">
      The original design used 6 API Gateway routes (<code>/dor</code>, <code>/dod</code>, etc.)
      mapped to one Lambda. Adding a new agent required a new route, a new service hook, and a code change.

      Tag-based routing eliminates this -- the Lambda scans work item tags to find the matching agent.
      Adding a new agent only requires Lambda env vars. No new route, no new hook.
    </ContentCard>

    <ContentCard icon="mdi:call-split" iconColor="bg-cyan" title="Why 2 Lambdas?">
      PR Router handles a fundamentally different event type (<code>git.pullrequest.comment-event</code>)
      with different gates (identity checking, bot loop prevention) and different queue parameters.
      Keeping it separate reduces complexity in both handlers.
    </ContentCard>
  </div>
</Section>

<SectionHeading
  badge="Libraries"
  badgeColor="info"
  title="Shared Libraries"
  subtitle="Deployed in the same zip as each Lambda -- reusable utilities for event processing."
  bg="primary"
/>
<Section>
  <div class="grid grid-cols-1 md:grid-cols-3 gap-6">
    <ContentCard icon="mdi:content-duplicate" iconColor="bg-amber-500" title="dedupe.js">
      DynamoDB conditional put for event deduplication. 1-hour TTL prevents duplicate webhook
      processing from ADO retry storms.
    </ContentCard>

    <ContentCard icon="mdi:speedometer" iconColor="bg-red-500" title="rate-limiter.js">
      DynamoDB atomic counters for per-pipeline and per-identity rate limits.
      Prevents runaway execution (max 20-30 runs per day per agent type).
    </ContentCard>

    <ContentCard icon="mdi:refresh" iconColor="bg-cyan" title="retry.js">
      Exponential backoff with jitter (3 retries). Used for ADO pipeline queue API calls
      and DynamoDB operations.
    </ContentCard>

    <ContentCard icon="mdi:key" iconColor="bg-emerald-600" title="aws-sig.js">
      AWS Signature V4 for ADO pipeline queue API calls. Handles auth header generation
      for cross-service communication.
    </ContentCard>

    <ContentCard icon="mdi:alert" iconColor="bg-red-500" title="dlq.js">
      Sends failed events to SQS Dead Letter Queue for later investigation.
      Ensures no events are silently dropped.
    </ContentCard>
  </div>
</Section>
