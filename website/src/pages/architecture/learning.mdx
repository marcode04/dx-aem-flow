  subtitle="Seven mechanisms already capture signals during execution. Most are per-session and lose memory between runs."

<Section>
  <div class="grid grid-cols-1 md:grid-cols-2 gap-6">
    <ContentCard icon="mdi:wrench" iconColor="bg-emerald-600" title="Self-Healing Loop">
      <strong>Scope:</strong> Per-session, per-step.
      dx-step-fix recovers from failures during execution (targeted fix + root cause escalation).
      Memory is lost after the session ends -- each run starts fresh.
    </ContentCard>

    <ContentCard icon="mdi:magnify" iconColor="bg-amber-500" title="Review-Fix Loop">
      <strong>Scope:</strong> Per-session, per-story.
      dx-step-verify runs up to 3 review-fix cycles. Same issues may recur across stories
      because findings are not persisted.
    </ContentCard>

    <ContentCard icon="mdi:calendar-week" iconColor="bg-cyan" title="Weekly Retrospective">
      <strong>Scope:</strong> Per-week, automation only.
      Written to S3 bundles. Human reviews retro and manually adjusts prompts/rules.
      Gap: only covers automation agents, not local CLI usage.
    </ContentCard>

    <ContentCard icon="mdi:test-tube" iconColor="bg-brand-700" title="Evaluation Framework">
      <strong>Scope:</strong> Pre-deployment quality gate.
      Fixture pass/fail results. Human updates fixtures when prompts change.
      Gap: does not learn from production runs.
    </ContentCard>

    <ContentCard icon="mdi:notebook" iconColor="bg-amber-500" title="Decision Journal">
      <strong>Scope:</strong> Per-run, automation only.
      Every agent decision logged with reasoning to S3. Human reviews journals and
      manually adjusts policy. Gap: no automated pattern extraction.
    </ContentCard>

    <ContentCard icon="mdi:shield" iconColor="bg-red-500" title="Policy Gate">
      <strong>Scope:</strong> Global, static.
      <code>pipeline-policy.yaml</code> restricts what each agent role can do.
      Manually maintained -- human updates after incidents.
    </ContentCard>

    <ContentCard icon="mdi:file-document" iconColor="bg-emerald-600" title="Shared Rules">
      <strong>Scope:</strong> Project-wide, git-tracked.
      <code>.ai/rules/</code> files read by dx skills and automation agents.
      Manual only -- no automated discovery or suggestion.
    </ContentCard>
  </div>

  <HighlightBox severity="warning" title="What's Missing">
    No per-project pattern accumulation across stories. No automated rule promotion. No calibration
    data for planning estimates. No cross-session memory for local CLI usage.
  </HighlightBox>
</Section>

<SectionHeading
  badge="Capabilities"
  badgeColor="success"
  title="Seven Learning Capabilities"
  subtitle="Each capability captures a specific signal type and feeds it back into future runs."
  bg="alt"
/>
<Section>
  <div class="grid grid-cols-1 md:grid-cols-2 gap-6">
    <ContentCard icon="mdi:wrench" iconColor="bg-emerald-600" title="1. Fix Pattern Memory">
      <strong>Storage:</strong> <code>fixes.md</code> + <code>fixes.jsonl</code>
      <br />After every dx-step-fix cycle, capture error type, message pattern, fix applied,
      and result. Before attempting a fix, dx-step-fix checks <code>fixes.md</code> for known patterns.
      If a strategy worked 3/3 times, try that first. Every 10 entries, <code>fixes.md</code> is regenerated.
    </ContentCard>

    <ContentCard icon="mdi:magnify" iconColor="bg-amber-500" title="2. Review Issue Tracker">
      <strong>Storage:</strong> <code>review-issues.md</code> + <code>review-issues.jsonl</code>
      <br />After every dx-step-verify or dx-code-reviewer run, capture issue type, file/component,
      auto-fix success, and cycle count. If 5+ issues of the same type appear, suggest adding a
      convention rule. Security patterns are promoted immediately (no threshold). Feeds into plan
      generation for preventive steps.
    </ContentCard>

    <ContentCard icon="mdi:hammer" iconColor="bg-red-500" title="3. Build Error Catalog">
      <strong>Storage:</strong> <code>patterns.md</code>
      <br />After every dx-step-build run, capture generalized error message patterns, affected file
      types/modules, root cause categories, and applied fixes. Common errors get fixed faster
      via known pattern matching.
    </ContentCard>

    <ContentCard icon="mdi:book-open-variant" iconColor="bg-cyan" title="4. Convention Discovery">
      <strong>Storage:</strong> <code>conventions-discovered.md</code>
      <br />After code review finds convention patterns, capture convention name, examples, and confidence
      level. Can be promoted to <code>.ai/rules/</code> when confirmed. Reduces false positive
      review findings over time.
    </ContentCard>

    <ContentCard icon="mdi:chart-line" iconColor="bg-brand-700" title="5. Execution Metrics">
      <strong>Storage:</strong> <code>metrics.md</code> + <code>runs.jsonl</code>
      <br />After every coordinator completes, capture ticket, flow, phases, steps, fixes, heals, tokens,
      duration, and result. Enables trend analysis, token budget planning, phase bottleneck identification,
      and estimate calibration.
    </ContentCard>

    <ContentCard icon="mdi:shape" iconColor="bg-cyan-dark" title="6. Story Archetype Matching">
      After each successful story completion, capture story type, component area, step count/types,
      and patterns needed. During planning, check for similar past stories to reuse step structure
      and predict fix/healing needs.
    </ContentCard>
  </div>

  <HighlightBox severity="info" title="7. Weekly Retrospective">
    On demand (<code>/dx-retro</code>) or automated. Reads all <code>.ai/learning/raw/&#42;.jsonl</code> files,
    aggregates success rates, common errors, and token trends. Identifies improving/degrading patterns.
    Suggests new rules and convention updates. Writes <code>.ai/learning/retro/YYYY-WNN-summary.md</code>.
  </HighlightBox>
</Section>

{`.ai/
  learning/
    fixes.md                    -- aggregated fix patterns
    review-issues.md            -- aggregated review findings
    patterns.md                 -- build error catalog
    conventions-discovered.md   -- discovered conventions
    metrics.md                  -- execution metrics summary
    retro/
      YYYY-WNN-summary.md      -- weekly retrospective
    raw/                        -- .gitignored
      runs.jsonl
      fixes.jsonl
      review-issues.jsonl`}
    </CommandBlock>
  </HighlightBox>

  <HighlightBox severity="success" title="Key Design Decisions">
    <ul class="text-sm space-y-1 mt-2">
      <li><strong>Skills decide where to write</strong> -- no central system</li>
      <li><strong>Threshold before promotion</strong> -- 3+ for build fixes, 5+ for conventions</li>
      <li><strong>Developer confirmation for rules</strong> -- never auto-create <code>.claude/rules/</code> files</li>
      <li><strong>Security patterns promoted immediately</strong> -- no threshold for secrets/vulnerabilities</li>
      <li><strong>Raw data is private</strong> -- gitignored, per-developer</li>
      <li><strong>Summaries are shared</strong> -- committable, team knowledge</li>
      <li><strong>Opt-in consumption</strong> -- skills check if learning files exist but work without them</li>
      <li><strong>No breaking changes</strong> -- learning is additive enhancement, not dependency</li>
    </ul>
  </HighlightBox>
</Section>
