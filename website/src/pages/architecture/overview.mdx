<SectionHeading
  badge="Subagents"
  badgeColor="error"
  title="Subagent Architecture"
  subtitle="Coordinator delegates to specialized subagents. Each runs in isolated context with the right model tier."
  bg="primary"
/>
<Section>
  <div class="grid grid-cols-1 md:grid-cols-2 gap-6">
    <HighlightBox severity="warning" title="Coordinator Never Sees File Contents">
      The coordinator only receives compact summaries -- file lists, status codes, and diffs.
      This keeps context windows small and costs low.
    </HighlightBox>

    <HighlightBox severity="info" title="Isolated Contexts">
      Each subagent runs in its own context. Skills with <code>context: fork</code> are automatically
      isolated -- no shared state leaks between steps.
    </HighlightBox>

    <HighlightBox severity="error" title="Agent Type Naming">
      Always use the full prefixed name: <code>dx-core:dx-code-reviewer</code>, NOT just
      <code>dx-code-reviewer</code>. The short name fails with "Agent type not found".
    </HighlightBox>

    <HighlightBox severity="success" title="Fork Auto-Isolation">
      Skills marked with <code>context: fork</code> spawn a new Claude instance. The parent skill
      continues after the fork completes, receiving only the summary.
    </HighlightBox>
  </div>
</Section>

<Section>
  <div class="grid grid-cols-1 md:grid-cols-3 gap-6">
    <ContentCard icon="mdi:star" iconColor="bg-red-500" title="Opus ($$$, slower)">
      Deep reasoning tasks that require understanding complex interactions across files and systems.

      - Implementation plans
      - Code review
      - Root cause analysis
    </ContentCard>

    <ContentCard icon="mdi:diamond" iconColor="bg-amber-500" title="Sonnet ($$, balanced)">
      Execution tasks that need good judgment but not maximum reasoning depth.

      - Step execution
      - PR review
      - AEM inspection
    </ContentCard>

    <ContentCard icon="mdi:circle" iconColor="bg-emerald-600" title="Haiku ($, fast)">
      Simple lookups and searches where speed matters more than depth.

      - File resolution
      - Doc search
      - Test running
    </ContentCard>
  </div>

<SectionHeading
  badge="Decisions"
  title="Design Decisions"
  subtitle="Key architectural choices and the reasoning behind them."
  bg="primary"
/>
<Section>
  <div class="grid grid-cols-1 md:grid-cols-2 gap-6">
    <ContentCard icon="mdi:puzzle" iconColor="bg-brand-700" title="Four Plugins, One Repo">
      Four independently installable plugins in a single monorepo. Non-AEM projects only need
      <code>dx-core</code>; multi-repo projects add <code>dx-hub</code>; AEM projects add <code>dx-aem</code>; any project wanting
      24/7 autonomous agents adds <code>dx-automation</code>. Single repo simplifies maintenance.
    </ContentCard>

    <ContentCard icon="mdi:cog" iconColor="bg-cyan" title="Config-Driven Over Hardcoded">
      All project-specific values read from <code>.ai/config.yaml</code> at runtime. The original
      skills had 50+ hardcoded values (org URLs, component paths, repo names). Replacing each with
      a config lookup makes skills reusable across any project.
    </ContentCard>

    <ContentCard icon="mdi:merge" iconColor="bg-emerald-600" title="Direct Skill Invocation">
      Coordinators invoke skills directly via <code>Skill()</code> calls instead of routing through
      an intermediary executor agent. Simpler architecture, fewer context switches, lower token cost.
    </ContentCard>

    <ContentCard icon="mdi:layers" iconColor="bg-amber-500" title="Three-Layer Override System">
      <code>file override &gt; config override &gt; plugin default</code>. Config overrides for
      quick tweaks, file overrides for deep customization, plugin defaults as sensible starting points.
      Projects can also shadow entire skills.
    </ContentCard>

    <ContentCard icon="mdi:eye-off" iconColor="bg-red-500" title="Skill Shadowing via Platform">
      Relies on Claude Code built-in skill precedence (project <code>.claude/skills/</code> &gt;
      plugin skills) rather than custom override logic. No custom code needed -- just create a
      file with the same name.
    </ContentCard>

    <ContentCard icon="mdi:currency-usd" iconColor="bg-cyan" title="Haiku for Lookup Agents">
      <code>dx-file-resolver</code>, <code>dx-doc-searcher</code>, and <code>aem-page-finder</code>
      use Haiku. These agents do simple lookups -- glob files, grep docs, call MCP search. Haiku is
      10x faster and cheaper than Sonnet with no quality loss.
    </ContentCard>

    <ContentCard icon="mdi:star" iconColor="bg-red-500" title="Opus for Code Review Only">
      Only <code>dx-code-reviewer</code> uses Opus. Code review is the one task where deep reasoning
      directly impacts quality (catching subtle bugs, understanding architectural implications).
      Other tasks are well-served by Sonnet.
    </ContentCard>

    <ContentCard icon="mdi:folder" iconColor="bg-brand-700" title="Spec Directory Convention">
      All per-ticket output goes to <code>.ai/specs/&lt;id&gt;-&lt;slug&gt;/</code> with predictable
      file names. Skills find each other's output by convention -- no data passing needed.
    </ContentCard>

    <ContentCard icon="mdi:school" iconColor="bg-emerald-600" title="Self-Learning in Coordinators">
      Learning (run logging, fix patterns, pattern promotion) lives in coordinator skills, not leaf
      skills. Coordinators see the full picture -- how many fixes, which heals worked. JSONL for
      machine data, promoted rules for proven patterns (3+ successes, 0 failures).
    </ContentCard>

    <ContentCard icon="mdi:console" iconColor="bg-amber-500" title="Scripts Stay Simple">
      Shell scripts for common operations, not Node.js or Python. Shared functions in
      <code>dx-common.sh</code> eliminated 26+ copies of duplicated helper scripts. No runtime
      dependencies -- works everywhere Claude Code runs.
    </ContentCard>
  </div>

  <HighlightBox severity="info" title="Automation as a Separate Plugin">
    The autonomous agent infrastructure (Lambda + ADO pipelines + AWS resources) lives in its own
    <code>dx-automation</code> plugin, not in <code>dx-core</code>. Interactive skills
    run when you ask and respond to you. Automation agents run 24/7 without you. They have completely
    different lifecycles -- separate plugin, separate install, separate operational skills.
  </HighlightBox>
</Section>
