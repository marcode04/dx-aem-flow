<SectionHeading
  badge="Layer 1"
  badgeColor="success"
  title="dx-step-fix -- First Responder"
  subtitle="One-shot fix attempt. Read the error, apply a minimal correction, re-verify, report."
  bg="primary"
/>
<Section>
  <div class="grid grid-cols-1 md:grid-cols-2 gap-6">
    <ContentCard icon="mdi:target" iconColor="bg-emerald-600" title="Strategy">
      - Read the blocked step's error from implement.md
      - Apply a minimal fix (not refactoring)
      - Re-run the step's test command
      - Success: mark step as <code>done</code>
      - Failure: mark step as <code>blocked</code> with diagnosis, STOP
    </ContentCard>

    <ContentCard icon="mdi:lightning-bolt" iconColor="bg-cyan" title="Superpowers Integration">
      Optionally invokes <code>superpowers:systematic-debugging</code> for structured 4-phase
      diagnosis (Observe, Hypothesize, Test, Fix) when the error is ambiguous or multi-layered.
      Falls back to inline diagnosis when superpowers is not installed.
    </ContentCard>
  </div>

  <HighlightBox severity="warning" title="One Attempt Only">
    dx-step-fix makes exactly ONE attempt per invocation. If the fix does not resolve the error,
    it marks the step as <code>blocked</code> and stops. The coordinator decides what happens next.
  </HighlightBox>
</Section>

<SectionHeading
  badge="Layer 2"
  badgeColor="warning"
  title="dx-step-fix (escalation) -- Deep Root-Cause Analysis"
  subtitle="After 2 consecutive fix failures, the healer takes over with extended thinking and a fundamentally different approach."
  bg="alt"
/>
<Section>
  <div class="grid grid-cols-1 md:grid-cols-2 gap-6">
    <HighlightBox severity="info" title="Type A: Step Blocked">
      Step has <code>Status: blocked</code> with a diagnosis. Heal creates ONE corrective step
      with distinctive numbering: <code>Step 3h</code> (first cycle), <code>Step 3h2</code> (second cycle).
      The corrective step MUST use a different strategy from the original.
    </HighlightBox>

    <HighlightBox severity="info" title="Type B: Review Failed">
      Full code review (dx-step-verify) failed after 3 cycles. Heal groups remaining Critical/Important
      issues by file and creates numbered corrective steps: <code>R1</code>, <code>R2</code>, etc.
      Second iteration uses <code>b</code> suffix.
    </HighlightBox>
  </div>

  <div class="grid grid-cols-1 md:grid-cols-3 gap-6 mt-6">
    <ContentCard icon="mdi:brain" iconColor="bg-amber-500" title="Extended Thinking">
      Uses ultrathink mode for deep reasoning about the root cause. The healer sees the full
      error context, previous fix attempts, and the original step intent.
    </ContentCard>

    <ContentCard icon="mdi:pencil-off" iconColor="bg-red-500" title="Never Writes Code">
      The healer NEVER writes source code directly. It only creates new steps in implement.md.
      The coordinator then executes those steps normally through dx-step.
    </ContentCard>

    <ContentCard icon="mdi:directions-fork" iconColor="bg-cyan" title="Different Strategy">
      If the first fix failed, the corrective step MUST use a fundamentally different approach.
      Returns <code>healed</code> (continue) or <code>unrecoverable</code> (stop).
    </ContentCard>
  </div>
</Section>

    <ContentCard icon="mdi:shield-check" iconColor="bg-red-500" title="Secret Scan (Phase 4)">
      IMMEDIATE STOP if secrets are found. No override, no retry, no healing. The pipeline
      halts and requires human intervention to remove the leaked secret.
    </ContentCard>

    <ContentCard icon="mdi:eye" iconColor="bg-brand-700" title="Code Review (Phase 6)">
      Uses dx-code-reviewer agent (Opus model). Confidence threshold of 80 on a 0-100 scale.
      Only reports issues the reviewer is CERTAIN about. Severity: Critical &gt; Important &gt; Minor.
      Review-fix loop runs max 3 cycles.
    </ContentCard>
  </div>

  <HighlightBox severity="success" title="Final Verification Gate">
    After the review-fix loop concludes, dx-step-verify optionally invokes
    <code>superpowers:verification-before-completion</code> for a final cross-cutting check --
    confirming all acceptance criteria are met before marking the step done.
  </HighlightBox>
</Section>

    <ContentCard icon="mdi:bug" iconColor="bg-red-500" title="systematic-debugging (dx-step-fix)">
      When the error is ambiguous or multi-layered, dx-step-fix optionally invokes
      <code>superpowers:systematic-debugging</code> for a structured 4-phase diagnosis:
      <ol class="mt-2 text-sm list-decimal list-inside space-y-1">
        <li><strong>Observe</strong> -- gather all error context</li>
        <li><strong>Hypothesize</strong> -- form candidate root causes</li>
        <li><strong>Test</strong> -- verify each hypothesis</li>
        <li><strong>Fix</strong> -- apply the validated correction</li>
      </ol>
      Falls back to inline diagnosis when superpowers is not installed.
    </ContentCard>

    <ContentCard icon="mdi:check-decagram" iconColor="bg-emerald-600" title="verification-before-completion (dx-step-verify)">
      After the review-fix loop concludes, dx-step-verify optionally invokes
      <code>superpowers:verification-before-completion</code> for a final cross-cutting check --
      confirming all acceptance criteria are met before marking the step done. This acts as a
      last safety net before commit.
    </ContentCard>
  </div>

  <HighlightBox severity="info" title="Soft Dependency">
    All superpowers hooks use a soft-dependency pattern: if the superpowers plugin is installed,
    the structured methodology is invoked via the Skill tool. If not installed, the skill falls back
    to condensed inline guidance. No configuration needed -- detection is automatic.
  </HighlightBox>
</Section>
