<SectionHeading
  badge="Architecture"
  badgeColor="secondary"
  title="Common Patterns"
  subtitle="Shared design principles across all coordinators."
  bg="primary"
/>
<Section>
  <div class="grid md:grid-cols-2 gap-4">
    <ContentCard icon="mdi:robot-off" iconColor="bg-brand-700" title="Coordinator-Only Architecture">
      All coordinators use <code>disable-model-invocation: true</code> -- they NEVER write code themselves.
      Every phase is delegated via direct <code>Skill()</code> invocation, keeping the main
      context lean (approximately 10 tokens per status update vs approximately 200 for full file reads).
    </ContentCard>
    <ContentCard icon="mdi:shield-check" iconColor="bg-cyan-dark" title="Isolated Context Management">
      Task tool calls use isolated subagents. Skills invoked via Skill tool have <code>context: fork</code>.
      The main orchestrator only sees compact return summaries, never reads spec files.
      Prevents context compaction in long-running pipelines.
    </ContentCard>
    <ContentCard icon="mdi:refresh" iconColor="bg-emerald-600" title="Idempotent Execution">
      Every skill checks if its output file already exists and is still valid before regenerating.
      Steps report one of: <strong>created</strong>, <strong>updated</strong>, or <strong>skipped</strong>.
    </ContentCard>
    <ContentCard icon="mdi:file-link" iconColor="bg-amber-500" title="Convention-Based Data Flow">
      No data is passed between skills. All skills find each other's output by predictable filenames
      in <code>.ai/specs/&lt;id&gt;-&lt;slug&gt;/</code>.
    </ContentCard>
    <ContentCard icon="mdi:wrench" iconColor="bg-red-500" title="Self-Healing Architecture">
      <strong>step-fix</strong> (first responder): one attempt to fix compilation/test/review failures.
      <strong>step-heal</strong> (deep analysis): creates corrective steps after 2 consecutive fix failures.
      <strong>3-cycle review loop</strong>: code review, auto-fix, rebuild, re-review (max 3 cycles).
      <strong>2 healing cycles max</strong>: if healing fails twice, stop for human intervention.
    </ContentCard>
    <ContentCard icon="mdi:source-branch" iconColor="bg-cyan" title="Cross-Repo Delegation">
      When <code>DX_PIPELINE_MODE</code> is set, code-writing coordinators check if the work targets
      another repo. If so, they write <code>delegate.json</code> and STOP -- the YAML step reads it
      and queues the equivalent pipeline in the target repo.
    </ContentCard>
  </div>
</Section>

    </ContentCard>

    <code>.claude/rules/learned-fix-&#42;.md</code> -- auto-loaded on every future run.
  </HighlightBox>

<SectionHeading
  badge="Deep Dive"
  badgeColor="warning"
  title="Phase Details"
  subtitle="Each phase of dx-agent-all explained."
  bg="alt"
/>
<Section>
  <div class="space-y-4">
    <ContentCard icon="mdi:clipboard-text" iconColor="bg-brand-700" title="Phase 1: Requirements (dx-req)">
      Subagent runs the full requirements pipeline: fetch, dor, explain, research, share.
      <strong>Interactive:</strong> Pauses for human review of dor-report.md and explain.md.
      <strong>Autonomous:</strong> Continues immediately.
      <strong>Fail condition:</strong> ADO fetch fails = ABORT.
    </ContentCard>

    <ContentCard icon="mdi:source-branch" iconColor="bg-cyan" title="Phase 1.5a: Cross-Repo Check (pipeline mode only)">
      Only runs when <code>DX_PIPELINE_MODE</code> is set. Reads <code>research.md</code> for cross-repo scope.
      If current repo is not the target: writes <code>delegate.json</code> and STOPS. Skipped in local mode.
    </ContentCard>

    <ContentCard icon="mdi:palette" iconColor="bg-amber-500" title="Phase 1.5 + 2: Figma (optional)">
      Scans raw-story.md for Figma URLs. If found, extracts design context (Phase 1.5) and generates a
      standalone HTML/CSS prototype using 2 parallel Haiku agents for convention discovery (Phase 2).
      <strong>Non-blocking:</strong> If Figma MCP fails, continues with a warning.
    </ContentCard>

    <ContentCard icon="mdi:map" iconColor="bg-brand-700" title="Phase 2b: Planning">
      Runs dx-plan, dx-plan-validate, and optionally dx-plan-resolve. Uses Opus with ultrathink
      for deep reasoning about dependencies.
      <strong>Fail condition:</strong> Validation FAIL = STOP.
    </ContentCard>

    <ContentCard icon="mdi:play" iconColor="bg-emerald-600" title="Phase 3: Execution (dx-step-all)">
      For each pending step: execute, test, review, commit. Self-healing: step-fix (1 attempt),
      step-heal (corrective steps, max 2 cycles). Commits deferred to after build verification.
    </ContentCard>

    <ContentCard icon="mdi:hammer" iconColor="bg-cyan-dark" title="Phase 4: Build Verification">
      Max 6 fix attempts. One error at a time, minimal fixes.
      <strong>Fail condition:</strong> Still broken after 6 = skip review/commit, continue to summary.
    </ContentCard>

    <ContentCard icon="mdi:shield-check" iconColor="bg-red-500" title="Phase 4.5: Full Code Review (dx-step-verify)">
      6-phase gate: compile, lint, test, secret scan, architecture, code review. Code review uses
      dx-code-reviewer (Opus, 80%+ confidence). Review-fix loop: max 3 cycles. If FAILED after 3 cycles,
      step-heal creates corrective steps, rebuilds, and re-reviews (max 2 healing cycles).
    </ContentCard>

    <ContentCard icon="mdi:git" iconColor="bg-brand-700" title="Phase 5a + 6: Commit + PR">
      Only if build AND review passed. Reads <code>auto-commit</code> and <code>auto-pr</code> from config preferences.
    </ContentCard>

    <ContentCard icon="mdi:file-document" iconColor="bg-amber-500" title="Phase 7: Documentation (optional)">
      Only if PR created. <code>aem-doc-gen</code> runs first (AEM projects), then <code>dx-doc-gen</code>
      reads its output and posts wiki page to sprint subfolder.
    </ContentCard>
  </div>
</Section>

<SectionHeading
  badge="Superpowers"
  badgeColor="secondary"
  title="Superpowers Integration"
  subtitle="Optional structured methodology hooks that enhance planning, debugging, and verification."
  bg="primary"
/>
<Section>
  <div class="grid md:grid-cols-3 gap-4">
    <ContentCard icon="mdi:lightbulb" iconColor="bg-brand-700" title="Design Exploration (Planning)">
      <code>dx-plan</code> supports an optional pre-planning step that invokes
      <code>superpowers:brainstorming</code> to surface alternative approaches before committing
      to an implementation strategy. Helps avoid tunnel vision on complex stories.
    </ContentCard>
    <ContentCard icon="mdi:shield-check" iconColor="bg-emerald-600" title="Verification Gate (Review)">
      After the review-fix loop concludes, <code>dx-step-verify</code> optionally invokes
      <code>superpowers:verification-before-completion</code> for a final cross-cutting check --
      confirming all acceptance criteria are met before marking the step done.
    </ContentCard>
    <ContentCard icon="mdi:bug" iconColor="bg-red-500" title="Systematic Debugging (Fix)">
      <code>dx-step-fix</code> optionally invokes <code>superpowers:systematic-debugging</code>
      for structured 4-phase diagnosis (Observe, Hypothesize, Test, Fix) when the error is
      ambiguous or multi-layered.
    </ContentCard>
  </div>

  <HighlightBox severity="info" title="Execution Methodology">
    Before Phase 1 begins, <code>dx-agent-all</code> runs an "Execution Methodology" setup step
    that optionally invokes <code>superpowers:executing-plans</code> to bootstrap structured
    plan-execution behaviour. All superpowers hooks fall back gracefully when the plugin is not installed.
  </HighlightBox>
</Section>

<SectionHeading
  badge="Reasoning"
  badgeColor="warning"
  title="Extended Thinking / Ultrathink"
  subtitle="Deep reasoning modes used in planning and healing phases."
  bg="alt"
/>
<Section>
  <div class="grid md:grid-cols-2 gap-4">
    <ContentCard icon="mdi:brain" iconColor="bg-brand-700" title="Planning Phase (Opus)">
      <code>dx-plan</code> and <code>dx-plan-validate</code> use ultrathink mode for deep reasoning
      about dependencies, step ordering, and risk assessment. This is the most token-intensive
      phase (≈80K tokens) but produces higher-quality plans with fewer downstream failures.
    </ContentCard>
    <ContentCard icon="mdi:medical-bag" iconColor="bg-amber-500" title="Escalation Phase (Opus)">
      <code>dx-step-fix</code> escalates to extended thinking for deep root-cause analysis when 2
      consecutive fix attempts fail. The escalated fix sees the full error context, previous fix attempts,
      and the original step intent to devise a fundamentally different approach.
    </ContentCard>
  </div>

  <HighlightBox severity="warning" title="Token Budget with Healing">
    The base token budget of ≈370K assumes no failures. With healing and fix cycles, a story can
    reach 500K-800K tokens. Each healing cycle adds ≈40-80K tokens for deep analysis plus the
    cost of executing corrective steps. The system trades tokens for autonomy -- most issues
    resolve without human intervention.
  </HighlightBox>
</Section>
