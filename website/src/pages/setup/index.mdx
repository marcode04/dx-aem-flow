<SectionHeading
  badge="Step 1"
  badgeColor="secondary"
  title="Install Plugins"
  subtitle="Add the shared marketplace and install the plugins your project needs."
  bg="primary"
/>
<Section>
  <CommandBlock label="Claude Code, GitHub Copilot CLI, or VS Code Chat">{`# Add marketplace (one-time)
/plugin marketplace add easingthemes/dx-aem-flow

# Install plugins
/plugin install dx-core@dx-aem-flow
/plugin install dx-aem@dx-aem-flow          # AEM projects only
/plugin install dx-hub@dx-aem-flow          # Multi-repo orchestration only`}</CommandBlock>

  <HighlightBox severity="info" title="Updating Later">
    <code>/plugin marketplace update dx-aem-flow</code> updates all installed plugins to the latest version.
    After upgrading, run <code>/dx-upgrade</code> to sync project files to the new plugin version.
  </HighlightBox>
</Section>

<SectionHeading
  badge="Step 2"
  badgeColor="secondary"
  title="Initialize & Configure"
  subtitle="Create project config, install convention rules, detect project structure."
  bg="alt"
/>
<Section>
  <HighlightBox severity="warning" title="AEM Projects — Set Environment Variables First">
    The AEM MCP server requires the <code>AEM_INSTANCES</code> environment variable. Set it <strong>before</strong> running
    <code>/aem-init</code> so component discovery can scan your running AEM instance for dialog fields,
    variants, and pages.
    <CommandBlock label="Shell profile (~/.zshrc or ~/.bashrc)">{`# Format: name:host:user:pass — comma-separated for multiple instances
export AEM_INSTANCES="local:http://localhost:4502:admin:admin"

# Multi-instance example (local + QA):
# export AEM_INSTANCES="local:http://localhost:4502:admin:admin,qa:https://qa-author.example.com:USER:PASS"

# Optional: accessibility testing + QA Basic Auth
# export AXE_API_KEY="your-axe-api-key"
# export QA_BASIC_AUTH_USER="your-qa-username"
# export QA_BASIC_AUTH_PASS="your-qa-password"`}</CommandBlock>
    Restart your terminal or run <code>{"source ~/.zshrc"}</code> after adding these.
    These env vars work in both Claude Code and Copilot CLI.
  </HighlightBox>

  <h3 class="text-lg font-semibold text-brand-900 mb-3 mt-6">Run these commands in order:</h3>

  <div class="grid md:grid-cols-2 gap-4">
    <ContentCard icon="mdi:numeric-1-circle" iconColor="bg-brand-700" title="/dx-init">
      Core initialization — creates <code>.ai/config.yaml</code>, convention rules, MCP config, agent templates, shared scripts. Interactive — asks about your Azure DevOps or Jira project, build commands, and base branch.
      If it detects an AEM project, automatically runs <code>/aem-init</code> to append the <code>aem:</code> section, install BE/FE rules by project role, and auto-generate <code>component-index.md</code> via AEM MCP.
    </ContentCard>
    <ContentCard icon="mdi:numeric-2-circle" iconColor="bg-cyan" title="/dx-adapt">
      Auto-detects project type (<code>aem-fullstack</code>, <code>aem-frontend</code>, etc.), toolchain (Node version, build tool, CSS compiler), and writes <code>project.type</code>, <code>project.role</code>, and <code>toolchain:</code> to config.
    </ContentCard>
  </div>

  <CommandBlock label="Setup">{`/dx-init                    # Core config + rules + MCP (chains to /aem-init for AEM projects)
/aem-init                   # Only if not already run as part of /dx-init
/dx-adapt                   # Project type + toolchain auto-detection`}</CommandBlock>

  <HighlightBox severity="info" title="/aem-init">
    <code>/dx-init</code> automatically chains to <code>/aem-init</code> when it detects an AEM project.
    Run <code>/aem-init</code> separately only if you skipped it during init or are adding AEM support to an existing project.
  </HighlightBox>

<SectionHeading
  badge="Step 3"
  badgeColor="secondary"
  title="Scan Your Project"
  subtitle="Teach the AI your actual conventions — replace generic placeholders with real values from your codebase."
  bg="primary"
/>
<Section>
  <p class="text-sm text-gray-200 mb-4">
    After init, your <code>.claude/rules/</code> files contain generic templates and <code>.ai/project/</code> seed data is minimal.
    <code>/dx-scan</code> reads your actual source code, discovers conventions interactively, and writes project-specific values.
  </p>

  <CommandBlock label="Run">/dx-scan</CommandBlock>

  <div class="grid md:grid-cols-3 gap-4 mt-6">
    <ContentCard icon="mdi:palette" iconColor="bg-cyan" title="Phase 1: Frontend">
      Discovers CSS variables, breakpoints, naming conventions (BEM/utility), component registration patterns,
      base classes, clientlib categories. Updates <code>fe-styles.md</code>, <code>fe-javascript.md</code>,
      <code>fe-components.md</code>, <code>fe-clientlibs.md</code>.
    </ContentCard>
    <ContentCard icon="mdi:code-braces" iconColor="bg-brand-700" title="Phase 2: Backend">
      Discovers Sling Model annotations, exporter patterns, dialog structure, HTL conventions,
      test framework, package structure. Updates <code>be-sling-models.md</code>, <code>be-components.md</code>,
      <code>be-dialog.md</code>, <code>be-htl.md</code>, <code>be-testing.md</code>.
    </ContentCard>
    <ContentCard icon="mdi:file-document" iconColor="bg-amber-500" title="Phase 3: Seed Data">
      Generates <code>architecture.md</code>, <code>features.md</code>, <code>file-patterns.yaml</code>,
      and <code>content-paths.yaml</code> in <code>.ai/project/</code>. Skips <code>component-index.md</code>
      if already created by <code>/aem-init</code>.
    </ContentCard>
  </div>

  <HighlightBox severity="info" title="Interactive Process">
    <code>/dx-scan</code> presents each discovery with concrete examples from your code and asks for confirmation
    before writing. You can correct, enrich, or skip any phase. This is a one-time step — run it once after
    initial setup, then only again if your project structure changes significantly.
  </HighlightBox>

  <h3 class="text-base font-semibold text-white mb-2 mt-6">Before vs After /dx-scan</h3>
  <div class="grid md:grid-cols-2 gap-4">
    <ContentCard icon="mdi:close-circle" iconColor="bg-red-500" title="Before (generic rules)">
      <code>fe-styles.md</code> says: "Use project CSS variables for spacing and colors." Agents don't know
      your actual variable names, breakpoints, or color tokens.
    </ContentCard>
    <ContentCard icon="mdi:check-circle" iconColor="bg-emerald-600" title="After (project-specific)">
      <code>fe-styles.md</code> says: "Use <code>$bp-mobile: 768px</code>, <code>$bp-tablet: 1024px</code>.
      Colors: <code>$color-primary: #1a1a1a</code>. Naming: BEM." Agents write code that matches your project.
    </ContentCard>
  </div>
