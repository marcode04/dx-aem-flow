<SectionHeading
  badge="Overview"
  title="What Gets Written to ADO"
  subtitle="Skills write three types of data to ADO: comments on work items, new work items (Bugs & Tasks), and wiki pages."
  bg="alt"
/>
<Section>
  <div class="grid grid-cols-1 md:grid-cols-3 gap-6">
    <ContentCard
      icon="mdi:comment"
      iconColor="bg-brand-700"
      title="Work Item Comments"
      tags={[{ label: '8 skills', color: 'primary' }]}
    >
      Structured markdown comments posted to stories and bugs. Each skill uses a unique
      signature tag (e.g., <code>[DoRAgent]</code>, <code>[DevPlan]</code>) for idempotent
      updates -- re-running a skill updates the existing comment instead of creating duplicates.
    </ContentCard>

    <ContentCard
      icon="mdi:clipboard-check"
      iconColor="bg-emerald-600"
      title="Work Item Creation"
      tags={[{ label: '4 skills', color: 'success' }]}
    >
      Creates child work items under stories: **Task** items for sprint planning
      and **Bug** items from QA verification. All created items are linked as
      children of the parent story.
    </ContentCard>

    <ContentCard
      icon="mdi:book-open-variant"
      iconColor="bg-cyan"
      title="Wiki Pages"
      tags={[{ label: '4 skills', color: 'info' }]}
    >
      Technical documentation and PR review reports published to the ADO Wiki under
      sprint-organized paths. Generated from spec files, retroactively from PRs, or
      from code review findings. Also supports Confluence.
    </ContentCard>
  </div>

  <HighlightBox severity="info" title="Safety Model">
    All ADO writes use <code>format: "markdown"</code> (never HTML).
    Comments are **append-only** -- ADO MCP doesn't support editing existing comments,
    so skills post update comments instead. Most skills ask for **user confirmation** before
    posting. Bug creation always requires human approval.
  </HighlightBox>
</Section>

<SectionHeading
  badge="Comments"
  badgeColor="info"
  title="Work Item Comments"
  subtitle="Each skill posts a structured comment with a unique signature tag. Signatures enable idempotent updates -- the skill checks for existing comments before posting."
  bg="primary"
/>
<Section>
  <div class="grid grid-cols-1 md:grid-cols-2 gap-6">
    <ContentCard
      icon="mdi:format-list-checks"
      iconColor="bg-amber-500"
      title="/dx-req (DoR phase)"
      tags={[{ label: '[DoRAgent]', color: 'warning' }, { label: 'dor-check.md.template', color: 'default' }]}
    >
      Definition of Ready checklist with interactive checkboxes. BAs can check items directly in ADO, then re-run DoR to update the scorecard. Three modes: full post, update, or skip if unchanged.
    </ContentCard>

    <ContentCard
      icon="mdi:chat"
      iconColor="bg-cyan"
      title="/dx-req (share phase)"
      tags={[{ label: '[DevPlan]', color: 'info' }, { label: 'share-plan.md.template', color: 'default' }]}
    >
      Non-technical implementation plan for the team. Covers what's being built, the approach, scope (S/M/L), assumptions, and open questions. Suitable for pasting into Teams or standup notes.
    </ContentCard>

    <ContentCard
      icon="mdi:clipboard-check-outline"
      iconColor="bg-emerald-600"
      title="/dx-req-dod"
      tags={[{ label: 'DoD Check:', color: 'success' }, { label: 'dod-summary.md.template', color: 'default' }]}
    >
      Definition of Done validation results. Checks PR status, approving reviews, resolved threads, child tasks, tests, build, secrets, and accessibility. Posts pass/fail verdict with score.
    </ContentCard>

    <ContentCard
      icon="mdi:file-document"
      iconColor="bg-cyan"
      title="/dx-agent-re"
      tags={[{ label: '[RE Agent]', color: 'secondary' }, { label: 're-summary.md.template', color: 'default' }]}
    >
      Requirements Engineering summary. Structured requirements spec with task breakdown, identified risks, and effort estimates. Posted after the RE agent analyzes the story.
    </ContentCard>

    <ContentCard
      icon="mdi:timer"
      iconColor="bg-amber-500"
      title="/dx-estimate"
      tags={[{ label: 'ai:role:estimation-agent', color: 'warning' }]}
    >
      Estimation results: hours per task, story points, affected AEM pages, and complexity breakdown. Uses an HTML comment signature for machine-readable detection.
    </ContentCard>

    <ContentCard
      icon="mdi:bug"
      iconColor="bg-red-500"
      title="/dx-bug-triage"
      tags={[{ label: '(clarification)', color: 'error' }]}
    >
      Clarification questions posted when a bug description is ambiguous. Asks for missing reproduction steps, expected behavior, or environment details to unblock investigation.
    </ContentCard>

    <ContentCard
      icon="mdi:check-decagram"
      iconColor="bg-brand-700"
      title="/dx-bug-verify"
      tags={[{ label: '[BugVerify]', color: 'primary' }]}
    >
      Bug verification evidence. Before mode: confirms the bug is reproducible on QA (screenshots, console errors). After mode: confirms the fix works on local AEM.
    </ContentCard>

    <ContentCard
      icon="mdi:handshake"
      iconColor="bg-cyan"
      title="/aem-qa-handoff"
      tags={[{ label: '[QAHandoff]', color: 'secondary' }, { label: 'qa-handoff.md.template', color: 'default' }]}
    >
      Structured testing steps for the QA team. Two modes: lightweight (reuses /aem-doc-gen output) or full (creates own test page + screenshots). Maps acceptance criteria to numbered testing steps with expected results.
    </ContentCard>
  </div>

  <HighlightBox severity="success" title="Comment Templates">
    Templates live in <code>dx-core/data/templates/ado-comments/</code>.
    Each template defines the markdown structure, placeholder tokens, and signature line.
    The signature line always ends with an ISO timestamp for traceability.
  </HighlightBox>
</Section>

  subtitle="Skills that publish technical documentation and PR review reports to the ADO Wiki (or Confluence)."
  bg="alt"

<Section>
  <div class="grid grid-cols-1 md:grid-cols-2 gap-6">
    <ContentCard
      icon="mdi:book-open-variant"
      iconColor="bg-emerald-600"
      title="/dx-doc-gen"
      tags={[{ label: 'wiki', color: 'success' }, { label: 'post-implementation', color: 'success' }]}
    >
      Generates technical documentation from completed spec files -- architecture decisions,
      usage guide, API changes. Posts to ADO Wiki under <code>&lt;wiki-doc-root&gt;/&lt;Sprint&gt;/&lt;id&gt;-&lt;slug&gt;</code>.
      The final step in the development pipeline (Phase 7).
    </ContentCard>

    <ContentCard
      icon="mdi:magnify"
      iconColor="bg-cyan"
      title="/dx-doc-retro"
      tags={[{ label: 'wiki', color: 'info' }, { label: 'retroactive', color: 'info' }]}
    >
      Generates documentation retroactively for stories that shipped without docs. Fetches the
      ADO story, finds linked PRs, searches the codebase for changes, and produces wiki-ready
      documentation. No spec files needed.
    </ContentCard>

    <ContentCard
      icon="mdi:comment-text"
      iconColor="bg-cyan"
      title="/dx-pr-review-report"
      tags={[{ label: 'wiki', color: 'secondary' }, { label: 'PR review', color: 'secondary' }]}
    >
      Generates a categorized report from a single PR review -- groups comments by category
      (accessibility, bug, functionality, regression), tracks patch resolution, and posts
      to ADO Wiki under <code>&lt;wiki-pr-review-root&gt;/&lt;ticket&gt;</code>.
    </ContentCard>

    <ContentCard
      icon="mdi:comment-text"
      iconColor="bg-amber-500"
      title="/dx-pr-reviews-report"
      tags={[{ label: 'wiki', color: 'warning' }, { label: 'batch', color: 'warning' }]}
    >
      Generates categorized reports for multiple PR reviews in batch -- lists reviewed PRs,
      generates a report for each, and posts all to ADO Wiki. Ideal for sprint review
      summaries and tech lead dashboards.
    </ContentCard>
  </div>

  <HighlightBox severity="info" title="Wiki Configuration">
    Wiki paths are configured in <code>.ai/config.yaml</code>:
    <br /><code>scm.wiki-doc-root</code> -- technical docs (used by <code>/dx-doc-gen</code>, <code>/dx-doc-retro</code>)
    <br /><code>scm.wiki-pr-review-root</code> -- PR review reports (used by <code>/dx-pr-review-report</code>, <code>/dx-pr-reviews-report</code>)
    <br />Both support Confluence as an alternative -- configured via <code>tracker.provider</code>.
  </HighlightBox>
</Section>
